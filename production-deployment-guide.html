Internal Reference — STC Cotyarn Payroll System

Production, Deployment & Post-Deployment Guide

A working reference for taking the payroll system live and keeping it healthy afterward. This is your first project being run for a real firm rather than yourself — the technical bar barely changes, but the operational bar (backups, access control, incident handling, compliance) goes up a lot, because real wages and real people's data now depend on this running correctly every month. This document is meant to be read once fully, then kept around as a checklist you come back to.

01The Mindset Shift

A personal project fails quietly — you notice, you fix it, nobody else is affected. A production payroll system fails loudly: someone doesn't get paid correctly, or a backup turns out not to exist when you needed it, or fingerprint data leaks and now there's a legal problem, not just a bug. Three things change:

  • You are the entire ops team. No SRE on call, no second engineer to sanity-check a risky deploy. That means you need lightweight versions of the safety nets a bigger team would have — checklists, backups, staging environments — because there's no one else to catch your mistake.
  • The data is sensitive by default. Bank account numbers, fingerprint templates, salary figures. This isn't a to-do app. Treat every table with PII as something that needs an access-control story, not just a CRUD endpoint.
  • Mistakes have a deadline. Payroll runs on a schedule. "I'll fix it next week" doesn't work if next week is payday and the system is down or the numbers are wrong.
Core principle
None of this requires enterprise tooling or a big budget. It requires consistent small habits — automated backups, a staging environment, a written checklist before every deploy — done every single time, not just when you remember.

02Pre-Launch Checklist

Before the first real payroll cycle runs on this system, confirm every item below. Treat this as a gate, not a suggestion.

  • Production, staging, and local environments are clearly separated — different databases, different .env files, never shared credentials.
  • No secrets (DB passwords, JWT signing key, Fernet encryption key) exist in source code or git history. Run a secret-scan before the first public commit if the repo will ever go public or be shared.
  • HTTPS is enforced everywhere, including the /iclock ADMS endpoint and the login route. No plaintext HTTP in production.
  • Row Level Security (or equivalent access control) is configured on every table containing employee, payroll, or biometric data — not left in the default "fully open" state.
  • Database backups are enabled and you have personally restored one into a scratch database to confirm the backup is real and usable.
  • Error tracking (Sentry) is wired up and actually sends you a notification — test this by triggering a deliberate error.
  • Every endpoint that mutates money or attendance data validates input server-side, not just in the React form.
  • A rollback plan for the first deploy is written down before you deploy, not improvised after something breaks.
  • You've run at least one full payroll cycle in parallel with the existing Excel process and the numbers match exactly.
  • HR staff who will actually use the system have clicked through it once, before go-live, not after.

03Environment Strategy

Three environments is the minimum for any system handling real money: local (your laptop), staging (a clone of production config, fake or anonymized data), and production (real data, real consequences). The rule that matters most: never point a local or staging frontend at the production database, even "just to check something quickly." That single habit prevents most of the worst accidental-deletion or accidental-edit incidents.

3.1Why staging matters even for a one-person team

Staging isn't about scale — it's about having somewhere to break things safely. Before running a new migration or deploying a payroll-engine change, run it against staging first with a copy of realistic (but de-identified) data. If something is going to go wrong, you want it to go wrong on a table nobody depends on.

Common mistake
Testing "carefully" directly in production because spinning up staging feels like overhead. It isn't — a free-tier Postgres instance and a second Railway/Vercel deployment target cost very little next to the cost of a payroll mistake reaching real employees.

04Secrets & Configuration Management

  • .env files are in .gitignore from day one — confirm this, don't assume it.
  • Secrets live in the hosting platform's environment variable store (Railway, Supabase project settings, etc.), never hardcoded, never in a Slack message or a comment.
  • The JWT signing secret, the Fernet key used to encrypt fingerprint templates, and the database password are three separate secrets — rotating one should never require touching the others.
  • If a secret is ever exposed (committed by accident, pasted somewhere it shouldn't be), rotate it immediately rather than hoping nobody noticed. Rotation is cheap; a breach investigation is not.
  • Use a least-privilege database role for the running application. The app's day-to-day DB user should not have the ability to drop tables or alter schema — reserve that for a separate migration-runner credential you use manually.

05Security Hardening

The current industry reference point is the OWASP Top 10 (2025 edition). It's worth reading the real list once, but here's how the top categories map onto a system like this one:

OWASP CategoryWhat it means for this system
Broken Access Control (#1)Every query must be scoped to org_id. A bug that lets one org's HR user read another org's employee data is the single most damaging class of bug in a multi-tenant system. Test this explicitly, not just by inspection.
Security Misconfiguration (#2)Debug mode off in production. CORS locked to your actual frontend origin, not *. RLS enabled with real policies, not left disabled "for now."
Software Supply Chain Failures (#3, new)Know what packages you depend on. Run pip list --outdated / npm outdated periodically. Don't add a dependency for a one-line utility you could write yourself.
Cryptographic Failures (#4)Passwords already go through bcrypt — good. Fingerprint templates are already Fernet-encrypted at rest — good, keep that key out of source control. Consider whether bank account numbers also deserve column-level encryption, not just DB access control.
Injection (#5)You're already using parameterized queries via asyncpg ($1, $2...) — never string-format SQL, ever, even for "just an internal admin script."
Insecure Design (#6)The validation gate before payroll runs (blocking on attendance exceptions) is exactly this principle applied correctly — fail closed, not open.
Authentication FailuresJWT expiry should be short enough that a stolen token isn't useful for long. Consider basic lockout/backoff after repeated failed logins on /auth/login.
Security Logging & Alerting FailuresLog who ran payroll, who edited a deduction, who overrode an attendance record, and when. These are the logs you'll need if a number is ever disputed.
Mishandling of Exceptional Conditions (new)If the validation gate check itself throws an error, the safe default is to block the payroll run, not silently let it through.

5.1Specific to the ADMS / fingerprint device endpoint

/iclock/cdata is intentionally unauthenticated (the device can't do JWT), which means it's a public endpoint by design. Mitigate this by validating the device serial number strictly against the devices table, rate-limiting that endpoint, and — if the mill's network setup allows it — restricting it to the device's known IP or routing it through a VPN rather than leaving it open to the whole internet.

06Database Management

6.1Migrations

  • Never edit a migration that has already been applied to production. Write a new one. Migration history is an append-only log.
  • Keep migrations small and reversible where possible. A migration that adds a nullable column is low-risk; one that renames or drops a column in active use is high-risk and should be done in two steps across two releases (add new, backfill, switch reads, then drop old).
  • Run every migration against staging first, on a realistic data copy, before running it against production.

6.2Backups — the 3-2-1 rule

Keep 3 copies of critical data, on 2 different storage types, with 1 copy stored somewhere other than your primary cloud provider. In practice for this project:

  1. Automated daily backups from your managed Postgres provider (confirm this is actually turned on — it's often opt-in).
  2. A periodic manual export (pg_dump) of at least employees, payroll_records, and payroll_periods, stored somewhere separate — a different cloud account, even just a local encrypted drive.
  3. If your plan supports it, enable Point-in-Time Recovery (PITR) — this is what saves you when someone fat-fingers an update rather than the whole database disappearing.
Non-negotiable
A backup you have never restored is not a backup — it's an assumption. Restore one into a throwaway database before you trust it. Do this once before go-live, and re-test every few months.

6.3Connection handling & indexing

  • Use connection pooling (PgBouncer, or whatever your managed provider includes) so a burst of requests doesn't exhaust available DB connections.
  • Make sure foreign keys and frequently-filtered columns (org_id, employee_id, period_id, date on attendance_daily) are indexed — at low employee counts this won't matter yet, but it costs nothing to do correctly now.

07Deployment Process

Even a minimal pipeline beats manual, ad-hoc deploys: lint → test → build → deploy, automated on every push to your main branch, even if it's just a GitHub Action. The goal isn't sophistication, it's consistency — the same steps run the same way every time, so a deploy is never "did I remember to run the tests this time?"

  • Tag every production release (git tags, e.g. v1.4.0) so you always know exactly what code is live.
  • Schedule risky deploys away from payday and away from end-of-month, when payroll is actively being processed.
  • Database migrations should stay backward-compatible with the previous app version for at least one release — add new columns before removing old ones, so a mid-deploy moment never has app code expecting a column that doesn't exist yet.
  • After every deploy, run a manual smoke test: log in, load the dashboard, hit one read endpoint that touches the DB. Thirty seconds of checking beats finding out from a user that something's broken.

08Testing Strategy

  • Unit tests for the payroll engine's three calculation paths (tier-based Labour, flat-rate Trainee, monthly Maintenance/Staff with OT/undertime) deserve the most attention of anything in this codebase — a bug here directly means a wrong paycheck.
  • Integration tests covering the full chain: punch logged → daily attendance processed → payroll run → payslip generated. This is where bugs hide that unit tests on individual functions won't catch.
  • UAT (User Acceptance Testing): before full cutover, have actual HR staff click through real workflows. They will find issues a developer testing their own code won't notice.
  • Parallel run: for at least one full payroll cycle, run the new system alongside the existing Excel process and compare every employee's net pay line by line. Don't fully retire Excel until the numbers match for a complete cycle.

09Monitoring, Logging & Alerting

  • Structured logs — include a request ID, org_id, and timestamp on every log line so you can trace one user's action through the system.
  • Error tracking — Sentry is already in your stack; confirm it actually notifies you (email or otherwise) rather than just collecting errors nobody looks at.
  • Uptime monitoring — a free external service (UptimeRobot, Better Uptime, etc.) pinging a /health endpoint every few minutes, so you find out the backend is down before HR does.
  • Business-level alerts, not just technical ones — for example, flag it if a payroll run produces ₹0 net pay for an unusual share of employees, or if attendance exceptions spike on a given day. These signal a logic bug, not a server crash, and technical monitoring alone won't catch them.

10Incident Response

Write a one-page plan before you need it, not during an outage. At minimum, answer:

  • If the system is down on payday, what's the fallback? (Likely: fall back to the Excel process for that cycle.)
  • Who needs to be told, and how, if employee data is exposed or the system is unreachable for an extended period?
  • Where do backups live, and do you remember the restore steps without having to relearn them under pressure?

After any real incident, write a short note — what happened, what you changed as a result. Five minutes now saves you from repeating the same mistake in six months when you've forgotten the details.

11Backup & Disaster Recovery

Two numbers worth defining, even informally:

  • RTO (Recovery Time Objective) — how long can the system realistically be down before it's a real problem? For a mill running payroll monthly, a few hours of downtime on a random Tuesday is very different from a few hours on disbursement day.
  • RPO (Recovery Point Objective) — how much data can you afford to lose if something goes wrong? For finalized payroll_records, this should be close to zero — once a payroll period is approved, that data needs to survive almost anything.

Beyond your cloud provider's backups, keep a periodic offline export of the most critical tables as a true "if the cloud account itself is compromised or deleted" copy.

12Documentation & Handoff

You're already doing the right thing instinctively by keeping a living handoff document as the project moves between tools and sessions — keep that habit permanently, not just during active development. At minimum, document:

  • How to run migrations, and how to roll one back if needed.
  • How to rotate each secret (JWT key, Fernet key, DB password) without breaking the running app.
  • How to restore from backup, written step by step, tested at least once.
  • Who has login access to what — the Supabase dashboard, the hosting platform, the production database directly.
Why this matters more here than usual
Right now you're the single point of failure for the firm's payroll system. If you're unavailable for a week, can someone else — even a future, more experienced version of you — figure out how to run payroll on time using only the documentation that exists?

13India Compliance — DPDP Act, 2023

India's Digital Personal Data Protection Act applies to digital personal data generally, and explicitly covers the categories this system stores: employee identification details, biometric data used for attendance, and financial information like bank account numbers and salary figures.

  • Legitimate use for employment: processing employee data for recruitment, attendance, payroll, and statutory compliance is generally permitted without separate explicit consent, because it falls under the Act's "legitimate use" exception for employment purposes. This exception does not extend to using the same data for anything unrelated to employment (e.g. marketing, third-party sharing) — that requires specific consent.
  • Biometric data gets extra weight: fingerprint templates should be collected only for a clear, documented purpose (attendance/payroll) and should not be retained indefinitely. Have an actual practice for deleting an employee's fingerprint template when they exit, rather than letting it sit in the database forever.
  • Data minimization: store what's needed for payroll and compliance, not more. Avoid collecting fields "just in case" they're useful later.
  • Breach awareness: know, even informally, what you'd need to do and who you'd need to notify if employee data were exposed. Penalties under the Act for serious data protection failures can be substantial, so this isn't a box-ticking exercise even at a single-mill scale.
  • Practical baseline: encrypt biometric data at rest (already done via Fernet), keep access logs of who viewed or modified sensitive records, define a retention period, and make sure you can actually delete an employee's data on request or on exit.

This is a working summary, not legal advice — for anything where you need a definitive compliance position (e.g. before scaling this into a product sold to other mills), it's worth a short consult with someone who practices data protection law in India.

14Versioning & Rollback

  • Tag every release that reaches production, even informally — you want to be able to answer "what code was running last Tuesday" without guessing.
  • Keep the previous deployable build around rather than deleting it immediately — most hosting platforms keep deploy history automatically; know how to redeploy a previous version in under a minute.
  • Before any risky schema change, take a manual backup snapshot in addition to your automated ones, specifically labeled with what's about to change.
  • Keep a simple running changelog — even a plain text file with dates and one-line summaries is enough to reconstruct "what changed and when" later.

15Performance & Scaling

For a single mill's employee count, performance will not be your limiting factor for a long time — correctness and reliability matter far more than speed right now. Don't over-engineer for scale you don't have yet. If this becomes a multi-tenant product sold to other mills later, that's the point to revisit: connection pool sizing under concurrent load, query plans via EXPLAIN ANALYZE on the heavier payroll queries, and pagination on every list endpoint that could realistically grow.

17Quick-Reference Checklists

A.Pre-Launch

  • Environments separated (local / staging / production)
  • Secrets out of code, in platform env vars
  • HTTPS enforced everywhere
  • Access control (RLS) configured on every sensitive table
  • Backups on, and restore tested at least once
  • Error tracking wired to actually notify you
  • Server-side validation on every mutating endpoint
  • Rollback plan written before first deploy
  • One full payroll cycle validated against Excel
  • HR staff have used the system before go-live

B.Launch Day

  • Deploy outside of payday / month-end if possible
  • Smoke test immediately after deploy (login, dashboard, one read endpoint)
  • Watch logs actively for the first few hours
  • Confirm the previous version is one click away if a rollback is needed

C.Weekly

  • Skim error tracking for new or recurring issues
  • Confirm automated backups actually ran

D.Monthly

  • Check for outdated dependencies
  • Review payroll run logs for anomalies
  • Update the handoff/documentation file with anything that changed

E.Quarterly

  • Backup restore drill
  • Access review — who can log into what
  • Rotate secrets if it's been a while or if anything was potentially exposed

18Glossary

TermMeaning
RTORecovery Time Objective — how fast the system needs to be back up after an outage.
RPORecovery Point Objective — how much data loss is tolerable if something fails.
CI/CDContinuous Integration / Continuous Deployment — automatically testing and deploying code on every push instead of doing it manually.
PITRPoint-in-Time Recovery — the ability to restore a database to its exact state at a specific past moment, not just to the last full backup.
RLSRow Level Security — database-enforced rules about which rows a given user/role is allowed to see or modify.
UATUser Acceptance Testing — real intended users (here, HR staff) testing the system before full go-live.
Smoke testA fast, shallow check after deploying that the basics still work — not a full test suite, just enough to catch an obviously broken deploy.
Zero-downtime deployA deployment strategy where the new version goes live without the application becoming unavailable, usually by keeping schema changes backward-compatible across the transition.
3-2-1 backup ruleKeep 3 copies of data, on 2 different types of storage, with 1 copy stored somewhere offsite from your primary provider.
