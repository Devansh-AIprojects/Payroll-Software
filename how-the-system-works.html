Internal Reference — STC Cotyarn Payroll System

How The System Actually Works

Most of this codebase was built by AI agents in Antigravity, in phases, across sessions you weren't fully present for. This guide walks through what each phase actually does — in the real code, not a summary of intentions — so that when something breaks, or someone asks how payroll gets calculated, or you want to improve a step, you know exactly where to look and why it's built that way. Read it once top to bottom, then keep it as a map.

01The Big Picture

Two independent things happen in this system, and almost every bug or question traces back to confusing them:

  1. The attendance pipeline — a physical fingerprint scanner pushes punch data to your server, which gets turned into daily attendance records, which get checked for problems.
  2. The payroll pipeline — once attendance for a month is clean, the payroll engine reads it and turns it into actual rupee amounts per employee.

The two pipelines are deliberately decoupled — payroll never touches raw punches directly, only the processed attendance_daily table. Here's the full chain:

router.py only handles HTTP concerns (auth dependency, request/response shape). service.py holds the actual logic and talks to the DB. queries.py is just raw SQL strings with $1, $2... placeholders — asyncpg fills them in, which is why SQL injection isn't a real risk here as long as this pattern is kept.

02Foundations — Auth, Database, Config

2.1How a request gets authenticated

Login (POST /api/v1/auth/login) checks the email against users, verifies the password with bcrypt, and issues a JWT containing user_id, org_id, and role, signed with secret_key and valid for 8 hours (matches a mill HR shift).

Every protected route depends on get_current_user, which decodes the JWT and then still queries the DB to confirm the user is still is_active. This means deactivating a user takes effect on their very next request — there's no caching to work around — at the cost of one extra DB query per request. require_hr and require_admin are just role checks layered on top of this.

Where the org boundary lives
Every query in this codebase filters by org_id pulled from the JWT, not from anything the client sends. This is the actual multi-tenant boundary — if you ever add a new query, this is the rule to never break.

2.2Database connection

database.py creates a single asyncpg connection pool (2–10 connections) at app startup and reuses it for the life of the process. get_connection() is for normal reads/writes; get_transaction() wraps multiple statements so they commit or roll back together — used for payroll status changes and the payroll run itself, where a half-applied result would be worse than an outright failure.

2.3Redis's actual role

Redis is used for exactly one active thing right now: a lock during payroll runs (key_payroll_lock(period_id), 1-hour TTL) so two people can't trigger payroll for the same period at once. Two other key builders exist in redis_client.py — a fingerprint-template cache key and a punch-dedup lock key — but neither is actually called anywhere in the service code yet. They're scaffolding for later, not bugs.

2.4Configuration

All secrets and environment-specific values come from .env via config.py (database_url, redis_url, secret_key, encryption_key, app_env, app_debug). is_production is just app_env == "production" and it controls two things in main.py: whether /docs and /redoc are exposed, and what CORS allows — more on that second one in Known Limitations, because it currently has a real problem.

03Phase 1 — Employee Master & Org Config

This phase is the "rules live in the database, not in code" foundation everything else depends on. modules/config exposes CRUD for shifts, categories, sub-categories, departments, salary components, and labour tier rates. modules/employees is the actual worker records.

3.1The category → sub-category → salary_type chain

Every employee belongs to a category (Labour / Maintenance / Staff) and a sub_category that carries a salary_type: tier, daily_flat, or monthly. This single field is what the payroll engine reads later to decide which of the three calculation paths to use — it's the hinge the whole payroll logic turns on.

_validate_refs() in employees/service.py enforces consistency at creation time: a tier or daily_flat employee must not have monthly_salary set, and a monthly employee must have it. Get this validation wrong on a new employee and the payroll engine will either silently use the wrong path or throw a confusing error a month later — this is the first place to check if a new hire's payroll looks wrong.

Files to know
  • app/modules/config/router.py / service.py — shifts, categories, departments, components, tier rates
  • app/modules/employees/router.py / service.py — employee CRUD + validation

04Phase 2 — Fingerprint Enrollment

Fingerprint templates are stored encrypted, never in plain form. enroll_fingerprint() takes a base64-encoded template from the request, decodes it, and re-encrypts it with Fernet (symmetric encryption, key = encryption_key from .env) before it ever touches the database. The API never returns the decrypted template back out — it's write-only from the API's perspective, which is exactly right for biometric data.

One fingerprint row per finger_index per employee. Re-enrolling the same finger uses PATCH, not a second POST — trying to enroll the same index twice via POST returns a 409 conflict on purpose.

Worth knowing
If encryption_key is ever lost or rotated without a migration plan, every previously enrolled fingerprint becomes undecryptable garbage — there's no recovery path coded for that. Treat this key with the same care as the JWT signing secret.
Files to know
  • app/modules/employees/service.py — _get_fernet, _encrypt_template, fingerprint functions

05Phase 3 — Devices & ADMS Punch Ingestion

The BioMax device doesn't poll your server — it pushes. adms_router is mounted at /iclock with no JWT auth, because the physical device can't carry a bearer token. It authenticates itself purely by sending its serial number (SN) as a query parameter, matched against the devices table.

5.1The two-call dance

  • GET /iclock/cdata — device boot/heartbeat. Server looks up the device by SN, touches last_seen_at, and replies with an options string containing ATTLOGStamp — the timestamp of the last punch we already have. This is what stops the device from re-sending its entire punch history every time it reconnects.
  • POST /iclock/cdata?table=ATTLOG — the actual punch data, plain-text, tab-separated lines like ATTLOG<uid><datetime><status><verify>. Parsed by _parse_attlog(), one line per punch.

5.2How a punch becomes a row

For each parsed punch: the device's internal uid is resolved to an employee_id via device_user_id (set once during enrollment). status maps to in/out (0/3/4 → in, 1/2/5 → out). verify determines matched — only fingerprint or face verify types count as matched; a password-based punch is stored but flagged unmatched. The insert uses ON CONFLICT (device_id, punched_at) DO NOTHING as the dedup guard, so a device resending old logs after a reconnect never creates duplicate rows.

Timezone, by design choice
All punch timestamps from the device are treated as IST directly (IST = UTC+5:30, hardcoded in service.py). This is intentional for now, not a bug — it's flagged in code comments as a future org-level setting, deferred because there's currently only one mill in one timezone.
Files to know
  • app/modules/attendance/service.py — handle_device_register, handle_attlog, _parse_attlog
  • app/modules/devices/ — device CRUD

06Phase 4 — Attendance Engine & Validation Gate

This is the most algorithmically dense part of the backend. process_daily_attendance() runs per date, per active employee, and does six things in sequence:

  1. Computes a shift window. _compute_shift_window() turns the employee's assigned shift (start time, end time, whether it crosses midnight) into an actual datetime range for that date, with a ±2 hour buffer on each side to catch early arrivals and late departures without bleeding into the next shift.
  2. Pulls matched punches inside that window.
  3. Pairs them. _pair_punches() takes the first IN and the last OUT in the window — so extra punches in between (breaks, accidental double-taps) don't break the pairing, they just get noted as an anomaly later.
  4. Calculates hours_worked from the gap between paired in/out, and derives status: absent if there are zero logs, present if hours clear half the shift's standard hours, half_day if below that. Important nuance: if only one side of the punch pair exists (IN with no OUT, or vice versa), status defaults to present, not absent — the reasoning is that a punch proves the person was physically there, so it's flagged as a data-quality exception for HR to verify rather than auto-marked absent.
  5. Calculates OT / undertime — but only for hours_based pay types (Maintenance, Staff). Labour and Trainee never get OT/undertime, full stop, regardless of hours worked.
  6. Detects exceptions via _detect_exception(), checked in priority order — first match wins: leave_conflict (leave exists but punches were still logged) → missing_punch → odd_punch_count (more than 2 punches that don't cleanly resolve to one IN/OUT pair) → excessive_duration (hours worked beyond shift length plus a 4-hour buffer).

The result is UPSERTed into attendance_daily — except rows that already have is_manual_override = TRUE, which are silently skipped so re-running the engine never clobbers an HR correction.

6.1The validation gate — enforced in the database, not the app

This is deliberate, and it's the strongest piece of "no hardcoded business logic" in the system: the rule that blocks payroll while exceptions are unresolved lives as a Postgres trigger (009_attendance_validation.sql), not as an if statement in Python. Every flagged record sets review_status = 'flagged'. When a payroll_periods row tries to transition draft → processing, trigger_block_payroll_processing() fires automatically and counts flagged rows for that org/month/year via fn_attendance_exception_count() — if the count is above zero, the UPDATE itself fails with a Postgres exception, no matter what code path tried to make it happen. Even a stray manual SQL update against the database would still be blocked.

Why this matters
Even if a future bug in the FastAPI app skips the application-level exception check in run_payroll(), the database itself still refuses to let the period move to processing. That's two independent layers protecting the same invariant — this is the gate actually doing its job, not just a UI warning.

HR resolves a flagged record one of two ways: Resolve (PATCH /attendance/daily/{id}/resolve) just clears the flag without touching the underlying data — for when the anomaly is actually fine (e.g. planned OT). Override (PATCH /attendance/daily/{id}) lets HR rewrite the status/hours/OT directly, sets is_manual_override = TRUE, and requires a override_reason.

Files to know
  • app/modules/attendance/service.py — process_daily_attendance, _detect_exception, override_daily, resolve_exception
  • db/migrations/009_attendance_validation.sql — the actual gate, lives in Postgres
  • app/modules/attendance/router.py — also defines leave_router for leave CRUD

07Phase 5 & 6 — Payroll Engine

engine.py is intentionally pure — no DB calls, no async, just Decimal math — which makes it the easiest part of the whole backend to unit-test and to trust. service.py is the orchestration layer that fetches data and calls into it.

7.1The three salary paths

PathWhoFormulaOT/UT?
A — tierLabour (Skilled) gross = daily_rate × days_present
daily_rate looked up by which bracket total days_present falls into		 No
B — daily_flatTrainee gross = flat_daily_rate × days_present No
C — monthlyMaintenance, Staff per_day = salary / 30
per_hour = per_day / standard_hours
gross = (per_day × days) + (per_hour × OT) − (per_hour × UT)		 Yes

Note: division by 30 is fixed regardless of how many days are actually in that calendar month — this is a deliberate industry-standard simplification, not a bug.

7.2Salary components — the configurable layer on top of gross

Once gross is known, apply_components() walks the org's configured salary_components list and computes each one:

  • percent_of_gross — straight percentage of gross (e.g. Basic = 50%, DA = 10%).
  • percent_of_component — percentage of a previously computed component, referenced by ref_component_id (this is how EPF = 12% of T Basic works — T Basic must be computed before EPF in calculation_order for this to resolve correctly).
  • fixed — flat rupee value.
Worth knowing — the EPF skip condition
Inside the loop: if comp_type == "deduction" and not epf_enrolled: continue. Today this is fine because EPF is the only deduction-type component configured. But read literally, it skips every deduction-type component for a non-enrolled employee, not just EPF specifically. If a second deduction component is ever added (e.g. a standing loan recovery line) that should apply regardless of EPF status, it would silently be skipped too for non-enrolled employees. Worth a closer look before adding any new deduction component.

7.3Running payroll, step by step

  1. Period must be draft or processing — anything else is rejected.
  2. Unresolved attendance exceptions for that month are checked at the application level too (in addition to the DB trigger) — any unresolved count blocks the run with a 400.
  3. A Redis lock is acquired on the period (payroll:lock:{period_id}) so two simultaneous run requests can't race each other; a second request gets a 409.
  4. If still draft, the period flips to processing — this is the exact moment the DB trigger gate from Phase 4 fires.
  5. Every active employee is processed independently. If one employee fails (e.g. no matching tier bracket found for their days_present), it's caught, logged into the run's errors list, and the loop continues — one bad record doesn't abort payroll for everyone else. Check the errors array in the run response after every run, not just the success count.
  6. For each employee: gross is calculated via the correct path, components are applied, old component rows are deleted and replaced (this is what makes re-running idempotent), and any existing manual deductions are re-summed into total_deductions and net_pay.
  7. The Redis lock is released in a finally block — even on error, the lock won't get stuck.

Re-running payroll for the same period (e.g. after fixing an attendance override) is safe — it recalculates everything from scratch but preserves manually entered deductions, because those live in a separate payroll_deductions table that the engine doesn't touch directly, only re-sums.

7.4Status workflow

draft → processing → approved → paid — strictly one-directional, enforced in _VALID_TRANSITIONS in service.py. Only draft → processing has an automated DB-level gate; processing → approved and approved → paid are pure human judgment calls with no automated check beyond "is this the legal next step." Deductions can only be added/removed while a period is draft or processing — once approved, the numbers are meant to be frozen.

Files to know
  • app/modules/payroll/engine.py — pure calculation, the actual math
  • app/modules/payroll/service.py — run_payroll, _process_single_employee, deduction + status logic
  • app/modules/payroll/router.py — endpoint surface

08Frontend Architecture

Vite + React, plain JavaScript (no TypeScript). App.jsx wraps every authenticated route in ProtectedRoute + a shared Layout shell (sidebar + page chrome). Routes mirror the backend modules directly: /employees, /attendance/process, /attendance/exceptions, /payroll/periods/:periodId, etc.

8.1The two-client split — and a gotcha bigger than documented

src/api/client.js exports two clients: api prepends /api/v1; raw calls the path exactly as given, no prefix. Looking at how main.py actually mounts routers:

Correction worth flagging
devices_router has no /api/v1 prefix — same situation as attendance. If any future frontend page calls device endpoints (e.g. a device management screen) through the api client by habit, it will silently 404 because it'll try /api/v1/devices instead of /devices. Same logic, easy to miss since attendance is the one everyone remembers.

8.2Auth on the frontend

AuthContext.jsx hydrates from localStorage on mount (token + cached user object), calls /api/v1/auth/login directly with raw fetch (not through the client wrappers), and stores access_token alongside a small user_id/org_id/role object. There's no token refresh logic — when the 8-hour JWT expires, the next API call will fail with 401 and the user needs to log in again. No silent refresh is currently wired up.

8.3How the trickiest pages actually work

AttendanceProcess.jsx has two independent halves on one page: a form that POSTs a date range to raw.post('/attendance/process', ...) to trigger the engine, and a separate employee picker that pulls that employee's attendance_daily rows to display. They don't talk to each other — running the engine doesn't auto-refresh the daily viewer below it.

Exceptions.jsx is the HR triage queue — month/year selector calls raw.get('/attendance/exceptions?...'), shows the flagged count prominently (turns green at zero, which is the visual cue that payroll can proceed), and offers Resolve or Override per row, matching the two backend paths described in Phase 4.

PeriodDetail.jsx is where the whole payroll workflow converges: it conditionally shows Run / Approve / Mark Paid buttons based on period.status, calls api.post('/payroll/periods/{id}/run') for the engine, and recomputes summary totals (gross/deductions/net) client-side from the records list rather than trusting a separate summary endpoint — so if those numbers ever look off, check that all pages of records were actually loaded, since this sums only the currently-fetched page, not the full period if pagination is in play.

09One Full Monthly Cycle, End to End

  1. Worker punches the BioMax scanner. Device eventually pushes the punch via POST /iclock/cdata; it lands in attendance_logs.
  2. HR opens Attendance Processing, picks the month's date range, clicks Process. process_daily_attendance() runs for every employee/date, writing attendance_daily rows and flagging anomalies.
  3. HR opens Exceptions, works through the flagged list — Resolve where the anomaly is acceptable, Override where the data needs correcting — until the flagged count hits zero.
  4. HR creates a Payroll Period for that month (if not already created) and clicks Run Payroll on the Period Detail page.
  5. The DB trigger silently confirms zero flagged exceptions as the period flips to processing; the engine calculates gross, components, and net pay for every active employee.
  6. HR reviews records, adds any manual deductions (advances, gifts, custom) per employee as needed — each addition recalculates that employee's net_pay immediately.
  7. HR clicks Approve (processing → approved) once satisfied — deductions can no longer be added or removed past this point.
  8. Payment is disbursed outside the system (bank/cash per employee's configured payment_mode), then HR clicks Mark Paid (approved → paid) to close the period.
  9. Individual Payslips are viewable any time after the run via the record/employee detail route.

13Known Limitations & Ideas To Improve

These came directly from reading the code, not speculation — ranked roughly by how much they matter.

Likely real production issue — CORS
In main.py: allow_origins=["*"] if not settings.is_production else []. Read carefully — in production this sets allow_origins to an empty list, which blocks every cross-origin request, not just untrusted ones. If the frontend is ever deployed on a different domain/port than the backend (which is the normal setup — e.g. frontend on Vercel, backend on Railway), every authenticated request from the browser will be rejected by CORS in production. Fix before going live: set allow_origins to the actual frontend's production URL, not an empty list.
EPF-skip condition is broader than its name suggests
Covered in detail in Phase 5/6 — apply_components() skips all deduction-type components, not just EPF, for non-enrolled employees. Fine today since EPF is the only one configured; worth revisiting before adding a second deduction component.
Intentionally deferred, not bugs
  • HALF_DAY_THRESHOLD_PERCENT (50%) and the IST timezone are hardcoded in attendance/service.py — both are explicitly flagged in code comments as future configurable settings, deferred on purpose since there's one org in one timezone right now.
  • exception_type's database constraint allows shift_mismatch and device_gap as valid values, but _detect_exception() never actually produces them yet — the schema is ahead of the detection logic. Room to extend later without a migration.
  • Redis fingerprint-cache and punch-dedup-lock key builders exist in redis_client.py but aren't called anywhere — scaffolding for a future optimization, not active.
No token refresh on the frontend
When the 8-hour JWT expires mid-session, the user just starts getting 401s with no automatic re-login flow or warning toast. For an HR shift-length token this is probably fine in practice, but a low-effort improvement would be a friendly "session expired, please log in again" redirect instead of silent failures.
Frontend totals are computed from the loaded page only
PeriodDetail.jsx sums gross/deductions/net from records in component state — if a period has more than 50 employees and pagination kicks in, the summary cards only reflect the currently loaded page, not the full period. Worth a dedicated summary endpoint if employee count grows past one page.
What's already done well, worth preserving
  • The validation gate living as a DB trigger rather than app logic — genuinely hard to accidentally bypass.
  • Idempotent payroll re-runs that preserve manual deductions — lets HR fix a mistake and re-run without fear.
  • Per-employee error isolation during a payroll run — one bad record doesn't take down the whole month.
  • Write-only fingerprint templates, encrypted before they ever reach the DB.

14Glossary

TermMeaning in this codebase
ADMSThe push protocol the BioMax device uses to send punch data to your server over HTTP, instead of you polling the device.
FernetA symmetric encryption scheme (from Python's cryptography library) used to encrypt fingerprint templates at rest.
JWTJSON Web Token — the signed, stateless credential issued at login, carrying user_id/org_id/role.
asyncpgThe async Postgres driver used directly (no ORM) — all SQL is raw, parameterized strings in queries.py files.
UPSERTINSERT ... ON CONFLICT ... DO UPDATE — insert a row, or update it if one already exists for that unique key. Used heavily so re-running processing/payroll is safe.
IdempotentRunning the same operation twice produces the same end result as running it once — true of both the attendance processor and the payroll engine by design.
DB triggerA function Postgres runs automatically on an INSERT/UPDATE — used here to physically block the payroll status transition, independent of the app code.
Redis lockA short-lived key set with NX (only-if-not-exists) used to prevent two concurrent payroll runs on the same period.
Path A/B/CThe three payroll calculation routes — tier-based, flat daily rate, and monthly-with-OT — chosen per employee by their sub-category's salary_type.
